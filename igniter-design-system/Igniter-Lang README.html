igniter-lang

A small language where every value carries a receipt — declared inputs, observable computation, and a result the compiler can prove.

declared · observable · time-aware

Igniter-lang turns a contract into a typed, observable program. You declare what goes in, what gets computed, and what comes out — the compiler checks the rest and hands you a result you can audit, replay, and trust.

No hidden state. No silent effects. Every computation is a node in a small dependency graph, and every node leaves a trace. That's the whole idea — and it fits on a napkin.

Quick taste

A contract is the unit of work. Here's the canonical one:

add.igcopy
-- The hello-world of igniter-lang
module Lang.Examples.Add

contract Add {
  input   a: Integer
  input   b: Integer

  compute sum = a + b

  output  sum: Integer
}

Types carry precision where it matters — money never rounds by accident:

bid_summary.igcopy
module SparkCRM.Finance

contract BidSummary {
  input   base_bid: Decimal[2]
  input   tax_rate: Decimal[4]

  compute gross_bid = mul(base_bid, tax_rate)

  output  gross_bid: Decimal[2]
}

Install

Igniter ships as a Ruby gem while it's pre-v1:

$ gem install igniter_lang --pre # compile a contract to a sealed .igapp $ igc compile add.ig --out add.igapp ✓ parsed · classified · typechecked · sealed ✓ add.igapp (2 inputs · 1 compute · 1 output)

Why igniter-lang

  • Declared, not discovered. Inputs and outputs are named up front. If it isn't declared, it can't happen.
  • Observable by default. Every computation is a node you can inspect — no stepping through a black box.
  • Time-aware. History and temporal types are first-class, so "what did this value look like last Tuesday" is a real question with a real answer.
  • Proof-carrying. Compilation produces evidence, not just a binary. The receipt travels with the result.
