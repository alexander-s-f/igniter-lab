## Why This Card

Repo-boundary P1 showed that `igniter-lab` has real architecture mixed with
generated output, stale stubs, proof artifacts, local IDE state, and maybe broken
fixture links. View-engine P2/P3 then corrected one tempting mistake:
`igniter-view-engine/out/` is generated, but **IDE-load-bearing**, so it must not
be cleaned blindly.

This card maps generated/stale/noise surfaces and identifies the smallest safe
cleanup candidates. It is a hygiene readiness card, not a deletion card.

## Authority

Readiness / cleanup map only.

Allowed:
- Inspect generated-output directories, ignored files, tracked/untracked state,
  symlinks, stale stubs, and test references.
- Produce one hygiene packet and close this card.
- Recommend exact later cleanup cards with candidate paths.

Not allowed:
- No `rm`, `git rm`, `mv`, `git mv`, copy, archive, or directory creation.
- No `.gitignore` changes.
- No Cargo/workspace changes.
- No code edits.
- No cleaning `target/`, `out/`, `build/`, `node_modules`, `.idea`,
  `.codex-local`, or proof outputs.
- No changing or restoring symlinks.
- No claim that generated == disposable unless current consumers are checked.

## Questions To Answer

### 1. What generated/noise classes exist?

Build an inventory by class:

```text
CARGO_TARGET       Rust target dirs
NODE_BUILD         node_modules/build/.svelte-kit/etc
GENERATED_OUT      out/ proof/runtime outputs
IDE_LOCAL_STATE    .idea/.vscode/.codex-local
EMPTY_STUB         empty or near-empty directories
STALE_SITE         doc/site placeholders with no active consumer
SYMLINK_OR_LINK    symlinks or expected links, present or absent
PROOF_OUTPUT       generated proof artifacts worth preserving or regenerating
UNKNOWN            needs human decision
```

For each class, include count, representative paths, tracked/ignored status, and
known consumers.

### 2. Which generated outputs are load-bearing?

Some generated output is disposable; some is runtime input today.

Special cases:

- `igniter-view-engine/out/` is generated and gitignored, but
  `igniter-ide` reads it at runtime. Mark as **LOAD_BEARING_GENERATED**.
- `igniter-compiler/igniter-view-engine/` was found as a 0-file stub. Confirm
  whether it remains a clean delete candidate.
- `target/` dirs are regenerated by Cargo but should not be deleted in this
  card.
- `igniter-ide/node_modules`, `.svelte-kit`, and `build` may be local build
  state. Confirm ignored/tracked status.
- `proofs/` and `out/` under historical proof trees may be evidence, not trash.

### 3. What is the `ln` situation?

P1/P2 found machine tests mentioning `../ln/...`, while `ln` appeared absent.

Answer:

- Does `igniter-lab/ln` exist now?
- Is it a symlink, directory, or absent?
- Which tests/docs reference `../ln`?
- Do those tests currently run or are they stale paths?
- What should own this next: generated-output hygiene, fixture-root cleanup, or
  machine test refactor?

No link creation or deletion in this card.

### 4. What is tracked vs ignored?

For each candidate cleanup path, classify:

```text
TRACKED_SOURCE          do not delete
TRACKED_PROOF_EVIDENCE  protect/archive deliberately
IGNORED_GENERATED       cleanup candidate, but check consumers
UNTRACKED_LOCAL         local state; cleanup maybe safe but not in this card
EMPTY_UNTRACKED_STUB    deletion candidate
UNKNOWN                 needs human decision
```

Use `git ls-files`, `git status --ignored`, and `git check-ignore` rather than
guessing from names.

### 5. What can be safely deleted later?

Make a candidate table:

- exact path;
- evidence;
- tracked/ignored/untracked status;
- active consumers;
- risk;
- proposed next-card owner.

Use cautious labels:

```text
SAFE_DELETE_CANDIDATE
ARCHIVE_CANDIDATE
KEEP_LOAD_BEARING
KEEP_EVIDENCE
NEEDS_HUMAN_DECISION
```

No deletion in this card.

### 6. What should not be cleaned?

Name paths that look noisy but must be protected, especially:

- `igniter-view-engine/out/` until IDE path/runtime expectations are refactored;
- tracked fixtures/proofs;
- any generated artifact that is used by tests or IDE tooling;
- local dev state if deleting it would disrupt active work without a clear win.
